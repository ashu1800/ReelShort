# Points Foundation Design

## 目标

建立阶段 1 的积分基础闭环：积分账户、积分流水、观看阶段奖励和 App 查询接口。后续充值、后台调整和权益能力都应在该账户/流水模型上扩展。

## 范围

本阶段实现：

- `GET /api/app/points/account`
- `GET /api/app/points/records`
- 观看进度达到 `25 / 50 / 75 / 100` 阶段时发放积分。
- 同一用户、同一剧集、同一分集、同一阶段只能发放一次。
- 积分账户余额只能通过积分流水变化。
- watch 上报响应返回本次新发放的奖励阶段和积分数量。

本阶段不实现后台积分调整、不实现充值订单、不实现支付、不实现可配置积分规则。每个观看阶段暂定奖励 `1` 积分，后续在运营配置模块中调整。

## 架构

`points` 模块包含账户、流水和观看奖励领取记录。`watch` 模块只负责记录观看进度，并在同一事务中调用 `PointsService.awardWatchProgress`，由 points 模块判断是否发放奖励。

核心表：

- `point_accounts`：用户积分账户，一名用户一条。
- `point_transactions`：积分流水，每一次余额变化必须有流水。
- `watch_reward_claims`：观看阶段奖励领取记录，使用唯一约束保证幂等。

## 奖励规则

后端根据 `WatchRecord.progressPercent` 判断已达到的阶段。比如 `76%` 会尝试发放 `25 / 50 / 75` 三个阶段中尚未领取的奖励。重复上报相同进度不会重复发放。

每个阶段当前奖励 `1` 积分。流水来源为 `WATCH_REWARD`，引用字段记录 `bookId`、`episodeNum` 和阶段。

## 错误语义

- 未登录访问积分接口：`401`。
- 用户已禁用：`403`。
- 积分查询为空时自动创建余额为 `0` 的账户。

## 测试

- 首次查询账户返回 `0`。
- watch 上报达到 `25%` 发放一次积分。
- 重复上报同一阶段不重复发放。
- 从 `0%` 直接上报 `80%` 会发放 `25 / 50 / 75` 三段。
- 积分流水与余额一致。
- 不同用户奖励隔离。
