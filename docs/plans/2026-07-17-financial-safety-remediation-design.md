# 资金安全全面修复设计

## 背景

当前测试阶段代码同时存在自动提现、VIP 链上收款、旧充值订单和积分账户等多条资金链路。审查确认了事务自调用失效、链上交易缺少持久化恢复、VIP 交易重放、客户端自定义充值兑换比例、管理员 2FA 覆盖、危险 Flyway 迁移、前后端契约漂移和 content-provider TLS 失败等问题。

本次允许清空并重建测试数据库。已经执行过的 V13-V19 不需要兼容现有测试数据，但仓库中的迁移必须改为对未来数据库安全、可验证且不清空资金数据。

## 已确认决策

- 保留服务端自动签名和自动广播能力。
- 热钱包私钥必须由后台管理页面逐次提交，不从环境变量或服务端密钥文件加载。
- 私钥只用于当前执行请求，不持久化、不记录日志、不进入响应。
- 不增加钱包换绑后的 24 小时提现冷静期。
- 银行卡占位表单和现有接口保持不变。
- 钱包绑定、更换和解绑需要当前密码验证。

## 提现与自动打款

提现状态机调整为：

`PENDING -> PREPARED -> BROADCASTED -> CONFIRMED`

异常状态分为 `FAILED_RETRYABLE` 和 `MANUAL_REVIEW`。只有链上达到配置确认数并验证执行成功后，才能把冻结积分正式扣除并将提现标记为完成。

管理员在后台选择提现单、输入对应链私钥和 TOTP 后提交执行请求。后端先验证管理员权限、TOTP、提现状态、私钥格式以及私钥派生地址是否等于配置的预期热钱包地址。预览接口不接收私钥。

后端在数据库事务中锁定提现单，为每笔打款创建唯一 payout attempt，分配 nonce 或链交易参数，生成已签名原始交易并保存确定性 txHash、nonce、网络、目标地址、金额、原始交易和状态。私钥不写数据库。事务提交后广播已签名交易。

后台确认任务负责：

- 重播仍处于 `PREPARED` 的同一已签名交易；
- 查询 `BROADCASTED` 交易的 receipt、执行结果和确认数；
- 确认成功后在事务内锁定积分账户，扣除冻结积分、记录流水并完成提现；
- 对明确失败的交易标记为可重试；
- 对 RPC 超时、链重组、nonce 冲突或结果不确定的交易进入人工核对，禁止自动生成新交易。

Ethereum nonce 使用数据库中的热钱包 nonce 状态串行分配，禁止查询失败后回退到 0。相同 signed raw transaction 和 txHash 可安全重播。TRON 交易过期后，必须先确认原 tx 不存在，才能由管理员重新提交私钥生成新交易。

TRON 签名改为协议要求的 `r + s + recoveryId`，广播使用正确的已签名交易格式。两条链都必须验证交易成功和确认数，而不是仅凭节点返回 txHash 判定成功。

## VIP 链上收款

每个用户只允许一个 `PENDING` VIP 订单。订单快照保存创建时的收款地址、基础价格、唯一尾差、最终应付金额、创建时间和到期时间。

自动确认必须验证：

- TRC20 USDT 合约正确；
- 收款地址等于订单快照；
- 金额与订单最终应付金额精确相等；
- 交易时间处于订单有效期内；
- 交易执行成功并达到确认数；
- txHash 尚未被任何订单消费。

数据库对已消费 txHash 建唯一约束，对待支付用户和待支付金额建立可执行的唯一约束。订单过期任务不依赖收款地址配置。管理员修改全局地址只影响新订单。

手工确认使用订单写权限、TOTP 和同一套链上验证逻辑，并记录独立管理员审计事件。授予 VIP 时锁定用户行，避免同一用户并发确认丢失已购买期限。

## 旧充值、钱包和管理员安全

旧充值订单创建接口不再接受客户端指定的 pointAmount。当前没有真实充值商品配置，因此创建接口明确返回不支持，历史订单和支付事件继续只读保留。

钱包绑定、更换和解绑请求增加当前密码，后端验证当前用户仍为 ACTIVE 且密码正确。ERC20 地址拒绝零地址；TRC20 继续做 Base58Check 校验。不增加换绑冷静期。

管理员登录在账号已启用 TOTP 时必须同时提交有效验证码。`/api/admin/2fa/enable` 只能首次启用，已有 2FA 时必须使用验证旧码和新码的 `/rebind`。VIP 确认/拒绝改用订单写权限。

## 积分账务

后台积分调整请求增加必填幂等键，数据库保证唯一。重复请求返回首次结果，不重复修改余额。

公平模式下，余额整数变化、积分流水、单集领取记录和 API 响应使用同一结算结果，确保单集记录可以解释账户余额变化。数据库为积分账户增加 `balance >= 0`、`frozen_points >= 0`、`frozen_points <= balance` 和小数余数范围约束。

所有提现状态转换、VIP 人工确认、2FA 变更和后台积分调整写管理员审计日志。日志不得包含私钥或已签名原始交易。

## Content Provider TLS

保留 DNS 解析和公网地址校验，但 HTTPS 连接必须同时满足：连接到校验后的固定 IP、TLS SNI 使用原始域名、证书按原始域名校验、HTTP Host 使用原始域名。重定向逐跳重复该过程。不得再用普通 requests 请求直接访问替换域名后的 HTTPS IP URL。

## 客户端和后台

Admin Web 只在执行打款时提交私钥，预览不提交。请求结束、关闭弹窗和路由切换时清空私钥。页面展示 payout attempt 状态、txHash、确认进度、失败原因和人工核对状态。

Android 同步钱包当前密码参数、VIP 订单快照字段和新增错误状态，并修复 V13 删除积分转账后遗留的测试夹具。银行卡入口保持现状。

## 数据迁移

V13 移除所有清空用户及资金数据的 DELETE 语句，不再依赖清库完成认证模型切换；历史手机号字段和不再使用的数据表可保留，避免破坏性升级。V18 移除“V13 已清空数据”的假设。V19 不再把 TRC20 地址直接标记为 ERC20。

新增后续迁移创建 payout attempt、热钱包 nonce、VIP 快照字段、唯一索引、管理员订单写权限、积分幂等键和账务 CHECK 约束。测试数据库在实施后清空并重新执行完整 Flyway 链。

## 错误处理与恢复

- 外部链请求不得包在数据库事务内。
- 广播超时默认视为结果未知，通过 txHash 查询或重播同一 signed transaction 恢复。
- 只有明确未上链或明确执行失败的 attempt 才允许重新生成交易。
- 数据库完成阶段失败时，由确认任务使用同一 txHash 重试账务完成事务。
- 批量执行逐笔返回结果，一笔失败不阻断其余提现，但不得把部分失败伪装为整体成功。

## 测试与验收

测试采用 TDD，覆盖：事务代理、状态机、进程崩溃恢复、重复广播、Ethereum nonce 并发、TRON 签名和广播格式、链上 revert、确认数、VIP txHash 重放、时间窗和地址匹配、尾差并发、同用户 VIP 并发续期、2FA 覆盖、权限、钱包密码、积分幂等、公平模式对账、PostgreSQL Flyway 迁移以及前后端 DTO 契约。

最终执行后端全量测试、Android `app-core` 测试与 Debug 构建、Admin Web 生产构建、content-provider 全量 pytest、发布质量基线，并检查工作区和暂存区 whitespace。
