# Android 使用体验加固设计

## 背景与目标

本次改动修复 2026-07-13 使用体验复审发现的问题，范围仅限 Android App：资金操作重复提交、危险操作确认、账户弹层可达性、大字体适配、成功/错误反馈语义、验证码倒计时、银行卡占位、输入键盘与密码可见性、横屏海报网格和海报无障碍语义。

## 方案比较

### 方案 A：局部状态与共享 UI 契约（采用）

在现有 `AppUiState` 中增加明确的提交状态、消息类型和短信成功触发器；Compose 层复用可滚动表单、确认弹层和自适应布局 helper。该方案改动集中、可测试，保持当前 `AppStateController` 架构。

### 方案 B：引入全局事件总线和导航框架

用一次性事件流承载消息、确认和页面跳转。扩展性更强，但会扩大本次修复范围，并增加事件丢失和生命周期处理复杂度。

### 方案 C：重做 Me 与资金流程

把每项资金操作改成独立页面和完整导航栈。体验上限最高，但工作量和回归面远超当前缺陷修复。

## 选定设计

### 提交安全

- `AppUiState` 增加账户操作提交状态；Controller 在账户操作入口原子拒绝重复请求，并在结束时释放状态。
- 所有账户表单按钮绑定提交状态，提交时禁用并显示明确进度。
- 钱包解绑、积分转账和提现在真正请求前显示确认弹层，确认内容包含目标、数量或操作后果。

### 弹层与自适应布局

- 账户操作 `ModalBottomSheet` 默认完全展开。
- `SheetForm` 改为带 `verticalScroll`、`imePadding`、`navigationBarsPadding` 的共享容器。
- Me 卡片去掉阻碍动态字体的固定高度，海报卡在大字体下降低覆盖文字密度。
- Home 和 Discover 的海报网格使用基于可用宽度的自适应列数，横屏显示更多列。

### 反馈与验证码

- 用 `UiMessage` 和 `UiMessageType` 区分 `SUCCESS`、`ERROR`、`INFO`，替代成功信息写入 `errorMessage` 的做法。
- 顶部消息条按类型使用语义色、图标和 TalkBack `liveRegion`。
- 钱包和改密验证码倒计时只在后端发送成功事件后启动，失败不锁定按钮。

### 表单与占位功能

- 银行卡入口保留为不可提交的“暂未支持”说明，不再收集姓名和卡号。
- `LoginTextField` 支持字段用途、对应键盘、IME action、autofill，以及密码显示/隐藏。
- 钱包地址、手机号、验证码、积分金额和密码分别使用正确的输入类型。

### 无障碍

- 海报图片在可点击海报卡内作为装饰，卡片本身提供一次完整的标题与集数描述，避免 TalkBack 重复播报。
- 消息条提供实时区域语义，禁用/提交状态由原生按钮语义暴露。

## 测试与验收

- 纯 Kotlin 测试覆盖重复提交拒绝、消息类型和短信成功触发器。
- JVM UI 契约测试覆盖确认规则、响应式列数、字段输入类型、银行卡不可提交、动态字体布局策略和无障碍描述。
- 运行 `:app-core:test`、`:app:testDebugUnitTest`、`:app:assembleDebug`。
- 在雷电模拟器验证正常字号、150% 字体、竖屏、横屏、Home、Discover、Me、钱包弹层和资金确认。
- 覆盖安装 APK，检查启动进程和 `FATAL EXCEPTION`。
